## Why

The project currently uses pnpm as both the package manager and script runner for the `web/` frontend. The goal is **runtime unification**: replacing pnpm with bun to use a single tool for package installation, script execution, and eventually test running — eliminating the separate Node.js + pnpm setup step from both local development and CI.

Concrete motivations:

- **Simpler CI setup**: the current `web-build` job requires three steps (pnpm/action-setup, actions/setup-node, cache config) to bootstrap the frontend toolchain. With bun, this collapses to a single `oven-sh/setup-bun` action.
- **Faster installs**: bun's install is significantly faster than pnpm (~5–10×), which compounds in CI where cold installs happen on every tag push.
- **Unified runtime**: `bun run <script>` can replace `pnpm run <script>` with no behavioural change; bun executes Vite, tsc, and ESLint without needing Node.js on PATH.
- **Simpler tooling surface**: one tool instead of Node + corepack + pnpm.

## What Changes

- **MODIFIED** `package.json` (root): `"packageManager"` field updated to `bun@1.3.13`; `"workspaces"` array added (`["web", "third_party/vscode-codicons"]`).
- **MODIFIED** `web/package.json`: `"packageManager"` field updated to `bun@1.3.13`.
- **NEW** `bunfig.toml` (root): opts into the human-readable text lockfile (`[install] save-text-lockfile = true`).
- **NEW** `bun.lock`: generated by `bun install`; replaces `pnpm-lock.yaml` as the committed lockfile.
- **DELETED** `pnpm-workspace.yaml`: workspace roots moved to `"workspaces"` in root `package.json`.
- **DELETED** `pnpm-lock.yaml`: superseded by `bun.lock`.
- **MODIFIED** `cmake/CronymaxApp.cmake`: `find_program(PNPM_EXECUTABLE pnpm)` → `find_program(BUN_EXECUTABLE bun)`; all `pnpm` invocations replaced with `bun` equivalents (`bun install --frozen-lockfile`, `bun run build`, `bun run typecheck`, `bun run lint`).
- **MODIFIED** `.github/workflows/release.yml` (`web-build` job): remove `pnpm/action-setup@v4` and `actions/setup-node@v4`; replace with `oven-sh/setup-bun@v2`; update all `pnpm` CLI calls to `bun` equivalents.
- **MODIFIED** `README.md`: replace all pnpm installation and usage references with bun equivalents.

## Non-goals

- **No runtime migration for tests.** Vitest continues to run as the test framework; `bun test` is not adopted. `bun run test` still invokes `vitest run`.
- **No change to Vite, TypeScript, or ESLint configuration.** Build toolchain stays identical.
- **No change to the vscode-codicons subpackage.** It is used purely for raw SVG imports (`src/icons/*.svg?react`); no build scripts from that package are invoked.
- **No change to C++ or Rust code.**
- **No change to CMake options or build targets** beyond the package manager executable reference.

## Impact

- **Tooling dependency change**: `pnpm` is no longer required; `bun ≥ 1.1` is required (for text lockfile support). README and CI updated accordingly.
- **Lockfile format**: `pnpm-lock.yaml` (YAML, lockfileVersion 9) → `bun.lock` (text, bun format). Git history for the lockfile is not continuous across this change.
- **CI**: `web-build` job loses two steps (pnpm action + setup-node) and gains one (`setup-bun`). Node.js is no longer installed in the `web-build` job. `native-arm64` and `native-x86_64` jobs are unaffected (they skip the web build via `-DCRONYMAX_BUILD_WEB=OFF`).
- **`workspace:*` protocol**: bun supports the `workspace:` protocol; `@vscode/codicons` in `web/package.json` resolves correctly.
- **pnpm `--filter` → bun `--filter` (`-F`)**: bun uses the same `--filter <package-name>` flag for workspace scoping. The filter values in `release.yml` (`cronymax-web`) carry over unchanged.
- **Local developer workflow**: developers install bun once (`brew install bun` / `curl -fsSL https://bun.sh/install | bash`); `pnpm` can be removed from the machine or left dormant.
