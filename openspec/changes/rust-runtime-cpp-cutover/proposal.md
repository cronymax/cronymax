## Why

`rust-runtime-migration` made the Rust workspace (`crony` + `crates/cronymax` + `crates/cronygraph`) the authoritative runtime for runs, agents, memory, reviews, capabilities, and persistence. The Rust side is feature-complete and tested end-to-end, but the C++ host and the renderer JavaScript still own a parallel, in-process execution path: `Space::agent_runtime`, `FlowRuntime`, the renderer ReAct loop in `web/src/agent_runtime/`, the host `event_bus` SQLite events table, and the per-Run JSONL trace writer. The migration only pays off once that parallel path is removed and every bridge handler forwards to the Rust runtime over GIPS. Until then the product carries two implementations of the same semantics with all the reconciliation bugs that implies.

## What Changes

- **NEW** `app/runtime_bridge/` C++ module: a `RuntimeProxy` that owns the GIPS client connection to the standalone `cronymax-runtime` child process. Mirrors the gips client envelope semantics (Mach ports / SOCK_SEQPACKET / named pipes), encodes `ClientToRuntime`/`RuntimeToClient` envelopes, and exposes a synchronous request/streaming-subscription API to bridge handlers.
- **NEW** runtime supervisor in `MainWindow`: spawns `cronymax-runtime` (built from `crony/bin/`) on app startup, performs the protocol handshake, restarts the child on crash, and propagates clean shutdown.
- **MODIFIED** every bridge handler in [app/browser/bridge_handler.cc](app/browser/bridge_handler.cc) for the `agent.*`, `flow.*`, `review.*`, `events.*`, `inbox.*`, `permission.*`, and `document.*` channels: forward to `RuntimeProxy` instead of invoking in-process orchestration. The handler signatures stay stable so the renderer side does not change shape mid-rollout.
- **BREAKING** Renderer agent runtime removed: [web/src/agent_runtime/loop.js](web/src/agent_runtime/loop.js), [web/src/agent_runtime/llm.js](web/src/agent_runtime/llm.js), and the renderer-side tool dispatcher are deleted. The agent panel becomes a pure projection of runtime-emitted events received via the existing bridge subscribe channel (now backed by GIPS).
- **BREAKING** `Space::agent_runtime` and `Space::flow_runtime` fields removed from [app/browser/space_manager.h](app/browser/space_manager.h). Space switching binds the active Space's runtime-side identifiers to the UI; the runtime owns all per-Run lifecycle.
- **BREAKING** Host event bus reduced to shell/UI metadata: the SQLite `events` and `inbox` tables in [app/event_bus/event_bus.cc](app/event_bus/event_bus.cc) drop all `agent.*`/`run.*`/`tool.*`/`review.*`/`document.*` rows. UI surfaces consume `RuntimeAuthority::run_history` and the runtime event subscription instead. JSONL trace writing under `<workspace>/.cronymax/flows/<flow>/runs/<run-id>/trace.jsonl` is removed; runtime persistence at `<app_data_dir>/runtime-state.json` is the single source of truth.
- **REMOVED** [app/agent/agent_runtime.h](app/agent/agent_runtime.h)/`.cc`, [app/flow/flow_runtime.h](app/flow/flow_runtime.h)/`.cc`, [app/flow/trace_writer.h](app/flow/trace_writer.h)/`.cc`, [app/document/reviewer_pipeline.h](app/document/reviewer_pipeline.h)/`.cc` (the in-process ReAct loop drivers). Their tests are deleted alongside.
- **NO COMPATIBILITY SHIMS**: per `rust-runtime-migration` design Decision 6, the cutover is a single landing — there is no "both paths active" mode. Each PR in the rollout either lands a forwarded handler or does not.

## Capabilities

### New Capabilities
- `runtime-bridge`: C++ host-side GIPS client (`RuntimeProxy`), runtime child-process supervisor, bridge-handler forwarding contract, and handshake/reconnect lifecycle.

### Modified Capabilities
- `multi-agent-orchestration`: tool dispatch, run lifecycle, review/approval flow, and event/trace surfacing move from in-process C++/JS execution to runtime-mediated execution over GIPS. Renderer is a pure projection layer; semantic state ownership lives entirely in the Rust runtime.
- `space-manager`: `Space` no longer owns `AgentRuntime`/`FlowRuntime`/`event_bus`. Space activation binds UI resources (browser views, terminal sessions) to runtime-managed run identifiers. `SpaceStore` retains tab/terminal-block metadata only; `agent_traces` and similar semantic tables are dropped.

## Impact

- **Affected C++**: [app/browser/bridge_handler.{h,cc}](app/browser/bridge_handler.cc), [app/browser/space_manager.{h,cc}](app/browser/space_manager.h), [app/agent/](app/agent/) (deleted), [app/flow/](app/flow/) (mostly deleted; `WorkspaceLayout` retained for authored artifacts), [app/event_bus/](app/event_bus/) (reduced), [app/document/reviewer_pipeline.{h,cc}](app/document/reviewer_pipeline.h) (deleted), [app/document/review_store.{h,cc}](app/document/review_store.h) (deleted — runtime owns reviews), [app/sandbox/permission_broker.{h,cc}](app/sandbox/permission_broker.h) (now invoked from a runtime capability adapter rather than an in-process tool registry).
- **Affected JS**: [web/src/agent_runtime/](web/src/agent_runtime/) deleted; [web/src/](web/src/) panels for agent/flow/inbox/review rewired to consume runtime events through the existing bridge subscribe channel.
- **New runtime dependency**: `MainWindow` must locate and exec the `cronymax-runtime` binary built from `crony/bin/`. Build system gains a step that copies the binary into the app bundle alongside the helper apps.
- **Persistence migration**: existing host-side `<workspace>/.cronymax/flows/<flow>/runs/<run-id>/state.json` files are read once on first launch and forwarded to the runtime as a one-shot import; subsequent state lives in `<app_data_dir>/runtime-state.json`. Authored artifacts (flow YAML, agent YAML, doc-type YAML, documents, document history) stay in the workspace.
- **Tests**: C++ unit tests for `agent_runtime`/`flow_runtime`/`reviewer_pipeline`/`trace_writer` deleted. New end-to-end test that boots the runtime child, drives a complete run from a renderer-shaped bridge call, and asserts events stream back. Existing Rust e2e (`crates/cronymax/tests/runtime_e2e.rs`) and GIPS transport (`crony/tests/gips_transport.rs`) tests remain as the runtime-side coverage.
- **Security**: capability invocation now goes runtime → C++ adapter rather than renderer/host → in-process broker. The C++ adapter still consults `PermissionBroker`, but the *decision to ask* originates from the runtime, eliminating the renderer's ability to bypass approval by calling the broker directly.
- **Rollout**: not staged. Per design Decision 6 the cutover lands as a coherent set of PRs gated on the GIPS client + supervisor + at least one forwarded handler being green; subsequent PRs convert remaining handlers and delete the legacy paths. No mixed-authority intermediate state ships to users.
